注意：git diff 命令只在修改文件后有效,如果将修改后的文件add或者commit了，那么执行git diff命令将啥也不显示。

有时，我们在修改文件后，发现文件被自己改乱了。那么能不能回退到之前的版本呢？当然可以。git log命令即可解决这个问题。

先回顾一下aaa.txt刚刚经历的几个版本

先执行git log看看会产生什么内容

细心的读者一定发现了，git log命令显示的内容是由近到远的。commit后有一连串很长的数字，称为coomit id，是根据SHA1算法计算出的十六进制数，具体有什么用，一会儿就知道了。

如果嫌上面的git log输出内容过多，下面有个精简版的。在git log后加入-- pretty=oneline即可。这样是不是清爽了许多？

想要回退版本，最起码要知道当前版本吧。在git中，用 HEAD 表示当前版本。所以9302204...开头的commit id后面就有个HEAD标志，表示的就是当前版本的信息。既然HEAD是表示当前版本，那么上个版本，上上个版本用什么表示呢？在git中，HEAD^表示上个版本，HEAD^^表示上上个版本，以此类推，如果有上上上个版本，那就是HEAD^^^。如果想要回退100个版本，岂不是要在HEAD后写100个^？当然可以这样写，但为了简便，我们可以写成HEAD~100。

此时再打开aaa.txt文件就会发现cat这个内容没有了。

回退到上个版本虽然成功了，但是如果又后悔了，想恢复到刚才有cat的版本怎么办？

这里有两种办法。

方法一：使用commit id还原

到这里，你应该知道之前commit后面的那一连串长数字是干嘛的吧，作用就是用来恢复文件。当然使用这种方法有个前提，那就是你没有关git窗口（如果关了窗口找不到commit id也别急，请往下参考第二种方法）。commit id后的长数字一般取前四到五位就行了，如果不嫌烦，尽量越长越好，但没必要。

此时再打开aaa.txt文件，发现cat又回来了。

方法二：使用git reflog命令还原（适用于忘记coomit id情况使用）

git reflog 把整个文件的git操作日志都查出来了，想要查看最新版有cat版本的文件，只要找到第三次coomit的值即可。所以重点是带coomit的后三行。经查，倒数第三行就是提交的日志。前面开头的字母和数字，就是commit id的前几位，我们直接复制拿来用即可。
